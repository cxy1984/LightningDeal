### 订单超时自动取消 实现细节

**技术方案**：RabbitMQ 延迟队列（`x-delayed-message` 插件）

**实现链路**：
```
下单成功
  ↓
SeckillOrderConsumer.handleSeckillOrder()
  → 创建订单（status=0）
  → 发送延迟消息（30min）
  ↓ 30分钟后
handleOrderTimeout(orderId)
  → orderService.cancelOrder(orderId)
  → Redis库存 +1
  → MySQL sold_stock -1
  → 用户购买计数 -1
  → 用户标记移除
  → 用户可重新抢购
```

**修复内容**：
- `cancelOrder()` 补充了 Redis 库存恢复和用户标记移除
- `refundOrder()` 同样补充了 Redis 库存恢复
- 限流 key 改为全局限流（按 activityId，原为按用户+活动）

### 1.2 Vite WebSocket 代理报错
- 现象：终端频繁输出 `[vite] ws proxy socket error: Error: read ECONNRESET`
- 原因：Vite 代理 `/ws` 到后端时，连接被重置（页面刷新、跳转时触发）
- **修复方案**（二选一）：

  **方案 A：前端直连后端 WebSocket（推荐）**
  ```js
  // vite.config.js - 去掉 ws 代理
  proxy: {
    '/api': {
      target: 'http://localhost:8080',
      changeOrigin: true
    }
    // 删掉 /ws 代理
  }
  ```
  ```js
  // stores/websocket.js - 直连后端
  // 开发环境直接连后端端口
  const wsUrl = process.env.NODE_ENV === 'development'
    ? `ws://localhost:8080/ws/seckill/result/${userId}`
    : `${protocol}//${window.location.host}/ws/seckill/result/${userId}`
  ```

  **方案 B：忽略错误（最低成本）**
  ```js
  // vite.config.js - 抑制 ws 代理错误日志
  server: {
    proxy: {
      '/ws': {
        target: 'ws://localhost:8080',
        ws: true,
        configure: (proxy) => {
          proxy.on('error', () => {});  // 忽略错误
        }
      }
    }
  }
  ```

## 三、详细任务说明

### P1-4：活动状态自动流转（当前最高优先级）

**现状**：数据库 `status` 字段不会自动更新。活动 1-5 结束时间已过但仍为 `status=1`（上架中）。

**需要做**：
```java
@Component
@Slf4j
public class ActivityStatusTask {

    @Scheduled(fixedRate = 60000) // 每分钟执行
    public void syncActivityStatus() {
        // 1. 查询所有 status=1 且 startTime < now 的活动 → 改为 status=2
        // 2. 查询所有 status=2 且 endTime < now 的活动 → 改为 status=3
        // 3. status=3 的清理 Redis 缓存（stock/sold/users/result）
    }
}
```

### P1-5：活动管理 API 完善

**现状**：删除时仅判断 `status >= 2`，但上架中的活动如果已到开始时间也应禁止删除

**需要修复**：
```java
// 删除时增加时间判断
boolean isRunning = activity.getStatus() >= 2 
    || (activity.getStatus() >= 1 && LocalDateTime.now().isAfter(activity.getStartTime()));
```

### P2-18：JWT 安全增强 — Redis 黑名单 + Refresh Token 双 token

**问题**：当前纯无状态 JWT，签发后无法主动失效。被劫持、改密码、踢人下线等场景无法处理。

**方案**：

```
登录
  ↓
签发 accessToken（短期，15min）+ refreshToken（长期，7天）
  ↓
前端存 localStorage：accessToken + refreshToken
  ↓
accessToken 过期 → 请求 /auth/refresh → 校验 refreshToken → 签发新 accessToken
  ↓
主动失效（改密码/踢人/后台操作）→ refreshToken 加入 Redis 黑名单
  ↓
下次换 token 时检查黑名单 → 拒绝 → 用户重新登录
```

**涉及修改**：

| 文件 | 说明 |
|------|------|
| `JwtUtil.java` | 新增 refreshToken 签发和校验方法，accessToken 有效期改为 15min |
| `SecurityConfig.java` | 新增 `/auth/refresh` 白名单 |
| `AuthController.java`（新增） | 提供 `/auth/refresh` 刷新接口 |
| `BlacklistService.java`（新增） | Redis 黑名单管理（add/check），TTL 与 refreshToken 一致 |
| `JwtAuthenticationFilter.java` | 校验时检查 accessToken 是否在黑名单中（秒杀等操作拒绝） |
| 前端 `api/index.js` | 响应拦截器：401 时尝试用 refreshToken 刷新，刷新失败才跳登录页 |
| 前端 `stores/user.js` | localStorage 同时存 accessToken 和 refreshToken |

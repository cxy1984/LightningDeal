# 订单超时自动取消 — 业务流程与代码修改说明

> 基于 RabbitMQ `x-delayed-message` 插件实现，下单后 30 分钟未支付自动取消订单，恢复所有资源。

---

## 1. 涉及文件

| 文件 | 修改类型 | 说明 |
|------|---------|------|
| `SeckillOrderConsumer.java` | 修改 | `handleOrderTimeout()` 消费延迟消息，调用 `cancelOrder()` |
| `SeckillOrderServiceImpl.java` | 修改 | `cancelOrder()` 补全 Redis 库存恢复 + 用户标记移除；`refundOrder()` 同步补全 |
| `SeckillController.java` | 修改 | 限流 key 改为全局限流（去掉 userId 维度） |

---

## 2. 完整业务流程

```
┌─────────────────────────────────────────────────────────────────────┐
│                     下单流程                                         │
└─────────────────────────────────────────────────────────────────────┘

用户点击秒杀
    │
    ▼
SeckillServiceImpl.seckill()
    ├─ RateLimitAspect 限流拦截（全局限流，按 activityId）
    ├─ Redis 预扣库存（DECR）
    ├─ 校验用户限购次数（Redis 计数）
    ├─ 记录用户到 set（seckill:users:{activityId}）
    └─ 发送消息 → seckill.exchange → seckill.queue
                            │
                            ▼
                ┌───────────────────────────┐
                │ SeckillOrderConsumer       │
                │ handleSeckillOrder()       │
                └───────────────────────────┘
                    │
                    ├─ Step 1: activityService.decrementDbStock()
                    │   → UPDATE sold_stock = sold_stock + 1
                    │      WHERE id=? AND sold_stock < total_stock
                    │   （MySQL 乐观锁兜底，如果库存不足则回滚 Redis 并结束）
                    │
                    ├─ Step 2: orderService.createOrder()
                    │   → INSERT 订单，status=0（待支付）
                    │
                    ├─ Step 3: WebSocket 推送成功结果
                    │
                    └─ Step 4: 发送延迟消息 → seckill.delay.exchange
                              → seckill.delay.queue
                              → setDelay(30 * 60 * 1000)   ← 30 分钟
```

```
┌─────────────────────────────────────────────────────────────────────┐
│                     超时取消流程（30 分钟后）                        │
└─────────────────────────────────────────────────────────────────────┘

seckill.delay.queue 中消息过期
    │
    ▼
┌───────────────────────────────────────────┐
│ SeckillOrderConsumer                       │
│ handleOrderTimeout(orderId)                │
└───────────────────────────────────────────┘
    │
    ├─ Step 1: orderService.getById(orderId)
    │   检查订单是否存在、是否仍为 status=0（待支付）
    │
    ├─ Step 2: 若 status=0 → orderService.cancelOrder(orderId)
    │   ┌──────────────────────────────────────────────┐
    │   │ cancelOrder() 内部（@Transactional）          │
    │   │                                              │
    │   │ ① UPDATE 订单 set status=2（已取消）           │
    │   │                                              │
    │   │ ② Redis 恢复：                                │
    │   │   decrement(seckill:user_count:{act}:{user})  │
    │   │     → 用户购买计数 -1（解除限购）               │
    │   │                                              │
    │   │ ③ MySQL 恢复：                                │
    │   │   activityService.revertSoldStock(activityId)  │
    │   │     → UPDATE sold_stock = sold_stock - 1       │
    │   │                                              │
    │   │ ④ Redis 恢复：                                │
    │   │   increment(seckill:stock:{activityId})        │
    │   │     → Redis 库存 +1（新：本次补充）             │
    │   │                                              │
    │   │ ⑤ Redis 恢复：                                │
    │   │   remove(seckill:users:{activityId}, userId)   │
    │   │     → 移除用户标记，允许重新抢购（新：本次补充） │
    │   └──────────────────────────────────────────────┘
    │
    ├─ Step 3: 若 status≠0 → 跳过（订单已支付/已取消等）
    │
    └─ Step 4: ack 确认消息
```

## 3. 修改细节

### 3.1 `SeckillOrderConsumer.java`

#### 3.1.1 `handleOrderTimeout()` — 超时取消消费者

**改动点**：

```java
// 原本：handleOrderTimeout 中重复执行了 cancelOrder 之外的资源恢复
orderService.cancelOrder(orderId);
activityService.incrRedisStock(order.getActivityId());              // ← 冗余，cancelOrder已有
redisTemplate.opsForSet().remove("seckill:users:"+..., userId);    // ← 冗余，cancelOrder已有
SeckillResult result = SeckillResult.fail("订单超时已取消", ...);
notifyUser(userId, activityId, result);                            // ← 冗余，WebSocket已有

// 改为：只调 cancelOrder
orderService.cancelOrder(orderId);
log.info("订单超时已取消 orderId={}, activityId={}, userId={}", ...);
```

**同时修复了异常处理**：

```java
// 原本：nack → 消费失败导致消息死循环重试
nack(channel, deliveryTag);

// 改为：ack → 异常时确认消息，避免死循环，人工介入
ack(channel, deliveryTag);
```

#### 3.1.2 `handleSeckillOrder()` — 发货延迟消息

**无实质性改动**，同段代码格式微调后功能不变。下单成功后向 `DELAY_EXCHANGE` 发送 `order.getId()`，设置 `setDelay(30 * 60 * 1000)`（30 分钟）。

### 3.2 `SeckillOrderServiceImpl.java`

#### 3.2.1 `cancelOrder()` — 取消订单

**改动点**（原方法缺少 Redis 库存回退和用户标记清理）：

```java
// 新增 ↓↓↓
// 恢复 Redis 库存（预减的库存退回去）
String stockKey = "seckill:stock:" + order.getActivityId();
redisTemplate.opsForValue().increment(stockKey);

// 移除用户标记，允许重新抢购
String userSetKey = "seckill:users:" + order.getActivityId();
redisTemplate.opsForSet().remove(userSetKey, order.getUserId().toString());
// 新增 ↑↑↑
```

**取消订单完整效果**：

| 资源 | 操作 | 效果 |
|------|------|------|
| MySQL 订单 | UPDATE status=2 | 标记为取消状态 |
| MySQL sold_stock | UPDATE sold_stock = sold_stock - 1 | 回退活动已售计数 |
| Redis seckill:user_count | DECR | 用户购买计数 -1，释放该用户的限购额度 |
| Redis seckill:stock | **INCR（新增）** | **恢复 Redis 预扣库存** |
| Redis seckill:users | **SREM（新增）** | **移除用户标记，允许重新参与** |

#### 3.2.2 `refundOrder()` — 退款订单

**改动点**（与 cancelOrder 保持一致）：

```java
// 新增 ↓↓↓
// 恢复 Redis 库存
String stockKey = "seckill:stock:" + order.getActivityId();
redisTemplate.opsForValue().increment(stockKey);
// 新增 ↑↑↑
```

### 3.3 `SeckillController.java`

**改动点**：

```java
// 原本：按用户+活动双维度限流
@RateLimit(
    key = "'seckill:' + #request.activityId + ':' + #authentication.principal"
)

// 改为：按活动全局限流（同一活动所有用户共享令牌桶）
@RateLimit(
    key = "'seckill:' + #request.activityId"
)
```

此举修复了压测中发现的问题：每个用户独立令牌桶导致多用户并发时限流失效。

## 6. 技术方案：RabbitMQ 延迟队列

采用 RabbitMQ 插件 `x-delayed-message`，而非 DLX+TTL 方案：

```
seckill.delay.exchange（x-delayed-message 类型）
    │  消息携带 x-delay header（30 分钟）
    │
    ▼
seckill.delay.queue
    │  30 分钟后消息变为 ready
    │
    ▼
handleOrderTimeout(orderId)
```

**配置（RabbitMQConfig.java）**：

```java
@Bean
public CustomExchange delayExchange() {
    Map<String, Object> args = new HashMap<>();
    args.put("x-delayed-type", "direct");
    return new CustomExchange(DELAY_EXCHANGE, "x-delayed-message", true, false, args);
}
```

```

```

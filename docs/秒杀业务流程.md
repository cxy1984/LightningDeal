# 🚀 LightningDeal 秒杀系统 — 完整业务流程

## 一、什么是秒杀？

**秒杀** 就是在极短时间内，大量用户抢购少量特价商品。

```
现实世界的秒杀：1000 个人冲进店里抢 100 台半价 iPhone
互联网秒杀：    10 万人在线点击"立即抢购"，只有前 100 人能买到
```

### 秒杀的核心挑战

| 挑战 | 说明 | 不处理的后果 |
|------|------|------------|
| **高并发** | 瞬间涌入成千上万请求 | 服务器宕机 |
| **超卖** | 卖出的数量超过库存 | 用户付了钱拿不到货 |
| **重复抢购** | 同一个人抢到多件 | 其他人买不到，不公平 |
| **公平性** | 脚本刷单 vs 真实用户 | 真实用户永远抢不到 |

### 本项目的设计目标

> **让 10 万人同时抢，系统不崩、库存不超、每人限购公平**

## 四、每个环节为什么这样设计？

### 1️⃣ 限流层 — 为什么在最前面？

因为限流是最轻量的检查，**不查数据库、不查 Redis**，只执行 Lua 脚本。

```
如果 10 万人同时请求：
  → 限流层挡住 99%，只放 1% 进去
  → 后面的数据库和 Redis 只承受 1% 的压力
  → 系统稳如泰山
```

**对比：不限流的情况**
```
10 万人同时请求
  → 全部打到 Redis 预减库存
  → Redis 撑不住 → 响应变慢
  → Tomcat 线程池占满 → 正常请求也进不来
  → 系统雪崩
```

### 2️⃣ 业务校验层 — 查什么？

三个问题，任何一个不通过就直接返回：

```
① 活动存在吗？          → SELECT * FROM seckill_activity WHERE id=?
② 时间对吗？            → now > startTime AND now < endTime？
③ 限购达到了吗？        → SELECT COUNT(*) FROM seckill_order
                          WHERE user_id=? AND activity_id=? AND status≠2
                          结果 ≥ limitPerUser → "已达到限购数量"
```

### 3️⃣ 重复检查层 — 为什么不用数据库？

```
用数据库查：
  SELECT COUNT(*) FROM seckill_order WHERE user_id=? AND activity_id=?
  → 每次都要查 MySQL，10 万人抢就 10 万次查询，MySQL 扛不住

用 Redis Set：
  SISMEMBER seckill:users:5 42
  → 内存操作，亚毫秒级，10 万次无压力
```

**但 Redis 有风险：** 重启/换库后数据丢失。
所以第 2 步的数据库限购检查是**兜底**，Redis 检查是**性能优化**。

### 4️⃣ 预减库存 — 秒杀的核心

这是整个系统**最关键的 1 毫秒**：

```lua
-- Redis Lua 脚本：原子操作，绝不超卖
local stock = redis.call('GET', KEYS[1])
if stock and tonumber(stock) > 0 then
    redis.call('DECR', KEYS[1])
    return 1   -- 减库存成功
else
    return 0   -- 没库存了
end
```

**为什么不能在 MySQL 里扣库存？**

```
MySQL 扣库存：
  UPDATE seckill_activity SET sold_stock = sold_stock + 1
  WHERE id = ? AND sold_stock < total_stock

问题：每次 UPDATE 都要加行锁 → 并发高的时候互相等待
      → InnoDB 行锁冲突 → 性能急剧下降
      → 10 万并发下单，MySQL 直接死锁

Redis 扣库存：
  单线程内存操作，O(1) 时间复杂度
  10 万并发也毫无压力
```

**Redis 和 MySQL 的双重保障：**

```
Redis 预减库存（第 4 步）：快速拦截，99% 的请求在这里被挡住
    ↓
MySQL 乐观锁扣减（MQ 消费者第 ① 步）：兜底，防止 Redis 数据不一致

两者都成功 → 才算真正抢到
任一失败   → 回滚，释放库存
```

### 5️⃣ 标记用户 — 为什么先标记后下单？

```
先标记：
  Redis Set 里加上用户 ID
  → 用户再点"抢购"，第 3 步直接拦住
  → 防止重复下单

如果下单失败怎么办？
  MQ 消费者里如果订单创建失败
  → 从 Set 里移除用户 ID（回滚）
  → 释放名额
```

### 6️⃣ 异步下单 — 为什么用 MQ？

```
同步下单（不用 MQ）：
  10 万人抢 100 件商品
  → 前 100 个请求扣库存成功
  → 100 个请求同时 INSERT 订单表
  → MySQL 写压力巨大，响应变慢
  → 用户要等好几秒才知道结果

异步下单（用 MQ）：
  前 100 个请求扣库存成功
  → 发 100 条消息到 MQ，立即返回"排队中"
  → MQ 消费者一个一个慢慢处理下单
  → 用户 1-2 秒后收到 WebSocket 推送结果
  → 体验流畅，系统稳定
```

**MQ 削峰的效果：**

```
                    │ 请求量
  10万 ┤            ▲
       │            █
       │            █
       │            █
       │   ┌────────────────┐
       │   │  限流挡住 90%   │
   1万 ┤   └────────────────┘
       │         █
       │         █
       │   ┌────────────────┐
       │   │  库存只有 100   │
   100 ┤   └────────────────┘
       │      ██
       │      ██
       │  ┌──────────────────┐
       │  │ MQ 削峰：每秒处理│
   10  ┤  │ 几十个订单       │
       │  └──────────────────┘
       └───────────────────────────► 时间
```

## 六、核心组件一览

| 组件 | 技术 | 作用 |
|------|------|------|
| **限流** | `@RateLimit` + Redis 令牌桶 | 削峰，控制请求速率 |
| **业务校验** | MySQL（MyBatis-Plus） | 活动存在、时间、限购 |
| **重复检查** | Redis Set | 快速拦截重复秒杀 |
| **预减库存** | Redis Lua 脚本 | 原子操作，防超卖 |
| **标记用户** | Redis Set | 记录已抢购用户 |
| **异步下单** | RabbitMQ | 削峰填谷，异步处理 |
| **数据库扣减** | MySQL 乐观锁 | 兜底防超卖 |
| **结果推送** | WebSocket | 实时推送秒杀结果 |
| **延迟取消** | RabbitMQ 延迟消息 | 30 分钟未支付自动取消 |
| **超时重试** | 死信队列 | 处理失败的消息 |

## 七、防超卖的三重保障

最坏情况分析：每一层都可能出问题，所以有三重保障。

```
第 1 重：Redis Lua 预减库存
  问题：Redis 宕机 → 预减没执行
  兜底：↓

第 2 重：MySQL 乐观锁扣减
  UPDATE ... WHERE sold_stock < total_stock
  问题：乐观锁失效（极少见）
  兜底：↓

第 3 重：业务层检查
  SELECT sold_stock, total_stock FROM seckill_activity WHERE id=?
  如果 sold_stock >= total_stock，拒绝下单
```

**三重保障同时失效的概率 ≈ 0**

## 八、总结

```
10 万人点击"抢购"
    ↓
限流挡住 9 万人 → HTTP 429
    ↓
剩下 1 万人进入业务校验
    ↓
活动已结束/限购已满 → 拦截 5000 人
    ↓
剩下 5000 人 Redis 预减库存
    ↓
库存只有 100 件 → 4900 人 "手慢啦"
    ↓
100 人进入 MQ 异步下单
    ↓
MQ 消费者一个个创建订单
    ↓
WebSocket 推送结果给 100 人
    ↓
100 人中 30 分钟内付款
    70 人超时未付 → 订单取消 → 库存释放
```

**核心思想：**
1. **层层削峰** — 每一层拦截大部分请求，最终只有少量请求打到数据库
2. **异步处理** — 秒杀结果不实时返回，MQ 削峰让系统平稳
3. **多重防超卖** — Redis + MySQL + 业务层三重保障
4. **公平限购** — 限流按用户维度，限购每人 N 件

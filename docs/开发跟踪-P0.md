## 🔧 第 2 轮修复（MQ 消费者消息卡住）

### 问题描述

MQ 消费者 `handleSeckillOrder()` 处理消息后，消息一直卡在"unacknowledged"状态，既不确认也不拒绝。导致队列积压，后续秒杀订单无法处理。

### 根因

`application.yml` 中配置了 `acknowledge-mode: manual`，即**手动确认模式**。在该模式下，Spring AMQP 不会自动发送 `basicAck`，必须由应用程序显式调用。

| 时间点 | 消费者代码 | ack 机制 | 结果 |
|--------|-----------|----------|------|
| 原代码 | 有 `@Transactional` | 事务提交时自动 ack | 正常 |
| 修复 `UnexpectedRollbackException` 后 | 去掉 `@Transactional` | **无 ack** | 消息永远卡住！ |

### 修复方案

三个 `@RabbitListener` 方法均改为手动 ack：

```java
@RabbitListener(queues = SECKILL_QUEUE)
public void handleSeckillOrder(SeckillMessage message, Channel channel,
        @Header(name = "amqp_deliveryTag") long deliveryTag) {
    try {
        // ... 业务逻辑 ...
        ack(channel, deliveryTag);  // 成功时确认
    } catch (Exception e) {
        nack(channel, deliveryTag); // 失败时拒绝（发到死信队列）
    }
}
```

### 涉及文件

| 文件 | 改动 |
|------|------|
| `SeckillOrderConsumer.java` | 三个 listener 添加 Channel + deliveryTag 参数，成功/失败后手动 ack/nack |

### 验证结果

- 50 并发请求 → 10 成功通过限流 → 10 个订单全部创建 → MQ 队列 0 条残留 ✅
- `seckill.dead.queue` 0 条（无异常消息） ✅

## 🔧 第 3 轮修复（活动删除增强 + Redis 清理 + 循环依赖）

### 问题描述

### 修复内容

```java
// deleteActivity() 增强
// 1. 清理 Redis：stock/sold/users/result 键
// 2. 级联取消所有待支付订单
// 3. 校验：已开始/结束的活动不可删除
```

### 循环依赖解决

`SeckillActivityServiceImpl` ↔ `SeckillOrderServiceImpl` 形成循环引用：

```
SeckillActivityServiceImpl
  → SeckillOrderService (新注入)
  → SeckillOrderServiceImpl
    → SeckillActivityService (原有)
    → SeckillActivityServiceImpl
```

**解决**：`@Lazy` 延迟注入 `SeckillOrderService`，打破构造器循环。

|------|------|

## ✅ P1-4：活动状态自动流转（定时任务）

**目标**：每分钟扫描数据库，自动更新活动状态，清理到期 Redis 缓存

### 后端

| 文件 | 说明 |
|------|------|
| `task/ActivityStatusTask.java` | 新增 @Scheduled 定时任务，每分钟执行 |

### 流转逻辑

```
每分钟执行
├─ status=1（上架中）+ startTime < now → status=2（进行中），预热 Redis
└─ status=2（进行中）+ endTime < now → status=3（已结束），清理 Redis 缓存
```

### 清理的 Redis 键
- `seckill:stock:{id}` — 库存
- `seckill:sold:{id}` — 已售
- `seckill:users:{id}` — 参与用户 Set
- `seckill:user_count:{id}:*` — 每人购买计数
- `seckill:result:{id}:*` — 秒杀结果
- `seckill:lock:{id}` — 分布式锁

### 验收标准
- [x] 即将开始的活动到时间自动变为进行中
- [x] 进行中的活动到结束时间自动变为已结束
- [x] 已结束活动的 Redis 缓存被清理
- [x] 不影响秒杀链路的实时时间校验（toVO + startTime/endTime 直接比较）

## ✅ P1-8：WebSocket 断线重连

**问题**：WebSocket 断开后永久失去推送，秒杀结果收不到

### 前端

| 文件 | 改动 |
|------|------|
| `vite.config.js` | 删除 `/ws` 代理，开发环境直连后端 `ws://localhost:8080`，避免 ECONNRESET |
| `stores/websocket.js` | 新增 `WSConnection` 类，支持指数退避自动重连 + 状态追踪 |
| `views/ActivityDetail.vue` | 重连后自动轮询拉取遗漏的秒杀结果 |

### 重连机制
```
断线 → onclose
  → 延迟 1s 重试第 1 次
  → 延迟 2s 重试第 2 次
  → 延迟 4s 重试第 3 次
  → ... 指数退避，最大 30s，最多 10 次
  → 重连成功 → 轮询兜底补遗漏结果
```

# LightningDeal 秒杀系统 — 项目经历

## 技术栈

**后端**：Spring Boot 2.7.18 / Java 8 / MyBatis-Plus 3.5.7 / Spring Security / Spring AOP  
**中间件**：MySQL 5.7 / Redis 7（Redisson 3.27） / RabbitMQ / Elasticsearch 7.17 + IK 分词 / MinIO  
**前端**：Vue 3 + Vite + Element Plus + ECharts + Axios  
**工具**：JJWT 0.12 / Swagger SpringDoc / EasyExcel / Hutool / Docker / Python 压测脚本

---

## 项目描述

LightningDeal 是一个面向高并发场景的秒杀系统，核心解决**瞬时流量冲击下的限流、库存防超卖、异步削峰**三大问题。系统采用 Redis 预扣库存 + Lua 脚本原子操作实现秒杀入口控制，结合 RabbitMQ 异步下单实现削峰填谷，MySQL 乐观锁兜底防止超卖，形成三层防护体系。同时集成 Elasticsearch 全文搜索、WebSocket 实时推送、RabbitMQ 延迟队列自动取消订单、实时数据大屏等配套能力，覆盖秒杀业务完整闭环。

---

## 个人贡献

### 1. 秒杀核心架构设计与开发

设计并实现了完整的秒杀核心链路：**限流 → Redis 预扣 → MQ 削峰 → 异步下单 → 状态推送**。

- **限流层**：基于 Redis 令牌桶算法实现全局限流切面（`@RateLimit` 注解），支持 SpEL 动态 Key，使用 Lua 脚本保证原子性，500 并发压测限流率 97.4%，P99 延迟 126ms。
- **库存层**：Redis DECR 预扣 + Lua 脚本原子校验限购次数 + MySQL 乐观锁兜底，确保库存零超卖。`cancelOrder()` 和 `refundOrder()` 同步恢复 Redis 库存、MySQL sold_stock、用户计数和参与标记，保证状态一致性。
- **异步层**：RabbitMQ 接收秒杀请求实现异步下单，消费者手动 ack 确保消息不丢失，死信队列兜底重试耗尽的消息。订单 30 分钟未支付通过 `x-delayed-message` 延迟队列自动取消并回退全部资源。

### 2. 搜索与实时能力建设

搭建了 Elasticsearch + IK 中文分词搜索系统，覆盖活动名称/商品名称/描述三个字段，支持搜索结果关键词高亮返回。

- **ES 同步**：DB 变更通过 MQ 异步同步到 ES（`TransactionSynchronizationManager` 确保事务提交后发送），3 次自动重试 + 死信队列不丢数据。
- **数据大屏**：基于 ECharts + WebSocket 轮询实现实时监控面板，展示 QPS 曲线（60s 环形缓冲区）、抢购流水（100 条最新记录）、商品销量排行榜、订单统计数据。限流切面每秒记录 QPS 到 Redis，定时任务聚合后供大屏展示。

### 3. 全栈业务闭环与质量保障

独立完成从数据库设计到前端交互的全栈开发，覆盖秒杀业务完整闭环。

- **全栈功能**：活动管理（CRUD + 图片上传至 MinIO + ES 同步）、用户中心（注册登录 + JWT 鉴权 + 个人信息修改）、订单系统（列表/详情/支付/取消/退款 + WebSocket 实时推送秒杀结果 + 30 分钟未支付自动取消）、管理后台（搜索/状态筛选/图片预览/活动状态只读保护）。
- **质量验证**：自研 Python 多用户并发压测脚本，模拟 500 并发真实秒杀场景，验证限流率 97.4%、库存零超卖、MQ 秒级消费、P99 延迟 126ms，并输出完整压测报告。
- **工程规范**：Git 提交粒度清晰（feat/docs/refactor 分类），开发跟踪文档完整覆盖每个功能的验收标准，技术债务和后续计划持续跟进。

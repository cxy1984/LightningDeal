# 认证鉴权执行流程详解

> 为什么 Security 能拦截、过滤器顺序、控制器获取用户的用途、放行/不放行示例

---

## 一、为什么 Security 能拦截请求？

### 本质：Java Web 过滤器机制

Spring Security 本质上是一堆 **Servlet Filter（过滤器）**，它们被注册到 Tomcat 的过滤器链中。

```
Tomcat 接收请求
  ↓
过滤器链（按顺序执行）
  ① 字符编码过滤器
  ② Spring Security 过滤器链（多个 Security Filter）
  ③ ...
  ④ DispatcherServlet（Spring MVC 入口）
```

Spring Security 通过 `@EnableWebSecurity` + `SecurityFilterChain Bean`，把自己的一大串过滤器（约 10~15 个）插入到 Tomcat 的过滤器链中。

**关键**：所有请求进来，**必须先通过 Security 过滤器链，才能到达你的 Controller**。

### 过滤器 vs 拦截器 vs AOP

| 机制 | 执行顺序 | 能拿到什么 | 谁在用 |
|------|---------|-----------|--------|
| **Servlet Filter（过滤器）** | 最早 | `HttpServletRequest`、`HttpServletResponse` | **Spring Security** |
| Spring Interceptor（拦截器） | 中间 | 同上 + Handler | 登录检查、日志 |
| AOP（切面） | 最晚 | 方法参数、返回值 | 限流、事务、日志 |

Security 选择 Filter 级别的原因是：**在请求到达 Spring MVC 之前就能拦截，不需要经过 Controller，性能更好也更安全**。

## 二、Spring Security 过滤器执行顺序

```
请求到达 Tomcat
  ↓
┌──────────────────────────────────────────────────────────────┐
│                  Spring Security 过滤器链                     │
│                                                              │
│  ① WebAsyncManagerIntegrationFilter                          │
│  ② SecurityContextPersistenceFilter       ← 从Session恢复上下文│
│  ③ HeaderWriterFilter                     ← 写安全响应头      │
│  ④ CorsFilter                              ← CORS 跨域       │
│  ⑤ CsrfFilter                              ← CSRF 防护       │
│  ⑥ LogoutFilter                            ← 登出处理        │
│  ⑦ JwtAuthenticationFilter（自定义）       ← ✅ 解析 Token   │
│  ⑧ RequestCacheAwareFilter                 ← 请求缓存        │
│  ⑨ SecurityContextHolderAwareRequestFilter                   │
│  ⑩ AnonymousAuthenticationFilter          ← 匿名用户         │
│  ⑪ SessionManagementFilter                ← Session管理      │
│  ⑫ ExceptionTranslationFilter             ← 异常翻译         │
│  ⑬ FilterSecurityInterceptor              ← ⭐ 权限裁决      │
└──────────────────────────────────────────────────────────────┘
  ↓
DispatcherServlet → Controller
```

### 第⑦步：JwtAuthenticationFilter（自定义）

```java
addFilterBefore(jwtAuthenticationFilter, UsernamePasswordAuthenticationFilter.class)
```

因为 `UsernamePasswordAuthenticationFilter` 不在链中（本项目未启用表单登录），实际上 `JwtAuthenticationFilter` 被插入到链中靠前的位置。

**它的工作**：
1. 从 Header 提取 Token
2. 验证签名
3. 把 `userId` 设到 `SecurityContextHolder`
4. `filterChain.doFilter()` 放行

### 第⑬步：FilterSecurityInterceptor（权限裁决器）

**这才是真正决定"放行还是不放行"的地方**。

它检查 `SecurityContextHolder` 中是否有 Authentication：

| SecurityContext 状态 | 请求路径权限配置 | 结果 |
|----------------------|-----------------|------|
| 有 Authentication | `permitAll()` | ✅ 放行 |
| 有 Authentication | `.authenticated()` | ✅ 放行 |
| **无 Authentication** | `permitAll()` | ✅ 放行 |
| **无 Authentication** | `.authenticated()` | ❌ **拦截 → 401** |

## 三、为什么要从控制器获取用户？

### 本质：知道"谁"在操作

```java
@GetMapping("/list")
public R<IPage<OrderVO>> list(Authentication authentication) {
    Long userId = (Long) authentication.getPrincipal();
    // ↑ 拿到当前登录用户的 ID

    // 用途1：只查自己的订单
    return R.ok(orderService.getUserOrders(userId, page, size, status));
}
```

### 核心用途

| 场景 | 没有 userId | 有 userId |
|------|------------|-----------|
| 查订单列表 | 查到所有人的订单 ❌ | 只查自己的 ✅ |
| 删除活动 | 任何人都能删 ❌ | 只有管理员能删 ✅ |
| 创建订单 | 不知道是谁的订单 ❌ | 关联到当前用户 ✅ |
| 支付 | 可以支付别人的订单 ❌ | 只能支付自己的 ✅ |
| 数据统计 | 无法按用户统计 ❌ | 可以统计每个用户的行为 ✅ |

### Spring 如何注入 Authentication？

```java
// 用户请求到达 Controller
// Spring 自动调用 SecurityContextHolder.getContext().getAuthentication()
// 拿到 JwtAuthenticationFilter 之前设置的对象
// 然后注入到方法参数中
public R<IPage<OrderVO>> list(Authentication authentication) {
    //                      ↑ Spring 自动注入
}
```

## 四、示例一：放行的请求（公开接口）

### 请求：登录

```
POST /api/user/login
Content-Type: application/json

{"username":"admin","password":"123456"}
```

### 执行过程

```
Tomcat 接收请求
  ↓
① WebAsyncManagerIntegrationFilter        → 放行
② SecurityContextPersistenceFilter        → 无 Session，跳过
③ HeaderWriterFilter                      → 放行
④ CorsFilter                              → 允许跨域，放行
⑤ CsrfFilter                              → 已禁用，放行
⑥ LogoutFilter                            → 不是登出路径，放行
⑦ JwtAuthenticationFilter                 → 没有 Authorization Header
                                              validateToken(null) → false
                                              **不设置 SecurityContext**
                                              filterChain.doFilter() 继续
⑧ RequestCacheAwareFilter                 → 放行
⑨ SecurityContextHolderAwareRequestFilter → 放行
⑩ AnonymousAuthenticationFilter          → 没有认证，创建匿名用户
⑪ SessionManagementFilter                → 放行
⑫ ExceptionTranslationFilter             → 放行
⑬ FilterSecurityInterceptor              → 检查路径权限：
                                              /user/login → permitAll() ✅
                                              **放行**
  ↓
DispatcherServlet
  ↓
UserController.login(Authentication authentication) {
    // authentication 是匿名用户，getPrincipal() 为 null
    // 但这里没有使用 authentication，而是直接接收请求体
    String token = userService.login(request);
    return R.ok(token);
}
```

### 关键点

- 没有 Token → `JwtAuthenticationFilter` 什么都不做
- `FilterSecurityInterceptor` 检查到 `/user/login` 是 `permitAll()` → **即使没有认证也放行**
- 登录接口不应该需要登录，这是合理的

## 五、示例二：不放行的请求（需要认证）

### 请求：查订单列表（没带 Token）

```
GET /api/order/list?page=1&size=10
（没有 Authorization Header）
```

### 执行过程

```
Tomcat 接收请求
  ↓
①~⑥ 同上，放行
  ↓
⑦ JwtAuthenticationFilter                 → 没有 Authorization Header
                                              validateToken(null) → false
                                              **不设置 SecurityContext**
                                              filterChain.doFilter() 继续
  ↓
⑧~⑪ 同上，放行
  ↓
⑫ ExceptionTranslationFilter             → 放行
  ↓
⑬ FilterSecurityInterceptor              → 检查路径权限：
                                              /order/list → 没有匹配 permitAll()
                                              .anyRequest().authenticated() → ❌
                                              **SecurityContext 中没有 Authentication**
                                              **抛出 AccessDeniedException**
  ↓
ExceptionTranslationFilter 捕获异常
  ↓
authenticationEntryPoint 处理             → 返回 401 JSON
```

### 返回结果

```json
{
  "code": 401,
  "msg": "未登录或 Token 已过期"
}
```

### 请求根本不会到达 Controller

```
FilterSecurityInterceptor 拦截
  ↓
ExceptionTranslationFilter 捕获
  ↓
authenticationEntryPoint 直接返回 401
  ↓
DispatcherServlet 收到的是 401 响应
  ↓
Controller 从未被执行
```

## 六、示例三：放行的请求（需要认证但带了 Token）

### 请求：查订单列表（带了 Token）

```
GET /api/order/list?page=1&size=10
Authorization: Bearer eyJhbGciOiJIUzM4NCJ9.eyJzdWIiOiIxIiwidXNlcm5hbWUiOiJhZG1pbiJ9...
```

### 执行过程

```
Tomcat 接收请求
  ↓
①~⑥ 放行
  ↓
⑦ JwtAuthenticationFilter                 → 提取到 Token
                                              validateToken(token) → true ✅
                                              解析出 userId=1
                                              new UsernamePasswordAuthenticationToken(1, null, empty)
                                              setDetails("admin")
                                              SecurityContextHolder.getContext().setAuthentication(...)
                                              filterChain.doFilter() 继续
  ↓
⑧~⑫ 放行
  ↓
⑬ FilterSecurityInterceptor              → 检查路径权限：
                                              /order/list → .anyRequest().authenticated()
                                              SecurityContext 中有 Authentication ✅
                                              **放行**
  ↓
DispatcherServlet
  ↓
SeckillOrderController.list(Authentication authentication) {
    // authentication.getPrincipal() = 1 (userId)
    // Spring 从 SecurityContextHolder 中取出 Authentication 注入
    Long userId = (Long) authentication.getPrincipal();  // = 1
    return R.ok(orderService.getUserOrders(1, 1, 10, null));
    //                                       ↑ 查到 userId=1 的订单
}
```

### 关键点

- `JwtAuthenticationFilter` 设置了 Authentication → `FilterSecurityInterceptor` 检查通过
- Controller 拿到 `userId=1`，只返回该用户的订单

## 七、对比总结

| 场景 | 是否带 Token | JwtAuthFilter 做了什么 | FilterSecurityInterceptor 裁决 | 结果 |
|------|-------------|----------------------|------------------------------|------|
| POST /user/login | ❌ | 不设 SecurityContext | `permitAll()` ✅ 放行 | ✅ 正常登录 |
| POST /user/register | ❌ | 不设 SecurityContext | `permitAll()` ✅ 放行 | ✅ 正常注册 |
| GET /activity/list | ❌ | 不设 SecurityContext | `permitAll()` ✅ 放行 | ✅ 看到活动列表 |
| **GET /order/list** | **❌** | **不设 SecurityContext** | **`.authenticated()` ❌ 拦截** | **❌ 401** |
| **GET /order/list** | **✅** | **设 SecurityContext** | **`.authenticated()` ✅ 放行** | **✅ 查到订单** |
| **POST /seckill/execute** | **❌** | **不设 SecurityContext** | **`.authenticated()` ❌ 拦截** | **❌ 401** |
| **DELETE /activity/1** | **✅** | **设 SecurityContext** | **`.authenticated()` ✅ 放行** | **✅ 执行删除** |

### 一句话总结

> `JwtAuthenticationFilter` 负责**认证**（你是谁），`FilterSecurityInterceptor` 负责**授权**（你能干什么）。认证通过 → 放行到 Controller；认证不通过 + 路径需要登录 → 直接 401，Controller 根本不会执行。

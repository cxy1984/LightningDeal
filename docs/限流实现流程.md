# ⚡ LightningDeal 秒杀限流实现流程

## 一、为什么需要限流？

秒杀系统的核心挑战是**瞬时高并发**。一个秒杀活动可能在 0.1 秒内涌入数千个请求，如果不加限制：

| 问题 | 后果 |
|------|------|
| 数据库被打爆 | MySQL 连接池耗尽，整个系统不可用 |
| 后端线程耗尽 | Tomcat 线程池满，正常请求也被拒绝 |
| 雪崩效应 | 一个接口被冲垮 → 拖垮整台服务器 → 拖垮上游网关 |
| 公平性丧失 | 脚本刷单者抢占所有名额，真实用户抢不到 |

**限流的目标**：在保证系统稳定的前提下，公平地分配秒杀名额。

```
限流 ≠ 拒绝用户
限流 = 削峰填谷，保证每秒进入后端的请求不超过系统承载能力
```

## 二、算法选型：令牌桶（Token Bucket）

本项目使用 **令牌桶算法**，相比其他算法的优势：

| 算法 | 特点 | 缺点 |
|------|------|------|
| **固定窗口** | 简单，1 秒内计数 | 窗口切换瞬间会冲两倍流量 |
| **滑动窗口** | 精度高 | 需要存储大量时间戳 |
| **漏桶** | 匀速流出，强制削峰 | 突发流量直接被砍，不够灵活 |
| **✅ 令牌桶** | **允许一定程度的突发**，平滑限流 | 需要维护桶状态 |

### 令牌桶原理

```
   ┌───────────────────────────┐
   │       令牌桶 (capacity=10) │
   │                           │
   │   ● ● ● ● ● ● ● ● ● ●   │  ← 初始满 10 个令牌
   │   ● ● ● ● ● ● ○ ○ ○ ○   │  ← 消耗了 5 个，剩 5 个
   │   ● ● ● ● ○ ○ ○ ○ ○ ○   │  ← 消耗了 6 个，剩 4 个
   │   ○ ○ ○ ○ ○ ○ ○ ○ ○ ○   │  ← 令牌消耗完，拒绝请求
   └───────────────────────────┘
           ↑ 每 1/qps 秒填充 1 个
```

**关键参数：**

| 参数 | 含义 | 示例值 |
|------|------|--------|
| **qps** | 每秒补充的令牌数 = 平稳状态下的 QPS 上限 | `5` → 平均每秒最多 5 个请求通过 |
| **capacity** | 桶的最大容量 = 允许的瞬时突发峰值 | `10` → 瞬间最多放行 10 个（但下一秒只能补充 5 个） |

**为什么这样设计？**

- 秒杀场景下，`capacity` 略大于 `qps`，允许瞬间并发进来几个请求，但后续立即被限流
- 填充速度（qps）决定了长期平均速率，容量决定了短期突发能力

## 四、代码实现详解

### 4.1 自定义注解 `@RateLimit`

```java
package com.lightningdeal.common.annotation;

@Target(ElementType.METHOD)
@Retention(RetentionPolicy.RUNTIME)
@Documented
public @interface RateLimit {

    int qps() default 100;         // 每秒填充令牌数
    int capacity() default 200;    // 桶最大容量
    String key() default "";       // 限流维度（SpEL 表达式）
    String message() default "请求过于频繁，请稍后重试";  // 提示文案
}
```

**三个核心属性：**

| 属性 | 作用 |
|------|------|
| `qps` | 控制**平均速率**。qps=5 表示平均每秒放行 5 个请求 |
| `capacity` | 控制**突发峰值**。capacity=10 表示瞬间最多放行 10 个 |
| `key` | 限流维度。支持 SpEL 从方法参数动态提取 |

### 4.2 实际使用（`SeckillController.java`）

```java
@PostMapping("/execute")
@RateLimit(
        qps = 5,                           // 每用户每活动每秒最多 5 次
        capacity = 10,                      // 每用户每活动瞬间最多 10 次
        key = "'seckill:' + #request.activityId + ':' + #authentication.principal",
        message = "秒杀太火爆啦，请稍后再试"
)
public R<SeckillResult> execute(@Valid @RequestBody SeckillRequest request,
                                Authentication authentication) {
    Long userId = (Long) authentication.getPrincipal();
    SeckillResult result = seckillService.executeSeckill(userId, request);
    return R.ok(result);
}
```

**Key 的设计 —— 按用户 + 活动维度限流：**

```
key = "seckill:" + activityId + ":" + userId
      ↓
实际 Redis key = rate_limit:seckill:1:42
                  ↑          ↑      ↑
             固定前缀    活动ID  用户ID
```

这意味着：
- **用户 A 抢活动 1** → 单独一个桶，qps=5，capacity=10
- **用户 A 抢活动 2** → 单独另一个桶，互不影响
- **用户 B 抢活动 1** → 单独一个桶，互不影响

> **为什么不使用全局 QPS 限制？**
> 秒杀场景下，每个用户限流比全局限流更公平。全局限流会误伤正常用户（"抢购总闸"），
> 而按用户维度限流确保每个用户公平竞争（"限购"）。

### 4.3 AOP 切面拦截（`RateLimitAspect.java`）

```
请求进入
    ↓
Spring AOP @Before 拦截所有带 @RateLimit 的方法
    ↓
① 解析 SpEL key → 从方法参数中提取 activityId 和 userId
② 构造 Redis key → "rate_limit:seckill:1:42"
③ 调用 Redis Lua 脚本（原子操作）
④ 返回值：
   1 → 放行，继续执行原方法
   0 → 抛出 RateLimitException(429)
```

#### SpEL 表达式解析

```java
private String parseKey(String keyExpr, JoinPoint joinPoint) {
    // 获取方法参数的名称（依赖 -parameters 编译选项）
    MethodSignature signature = (MethodSignature) joinPoint.getSignature();
    Object[] args = joinPoint.getArgs();

    // Spring 方法执行上下文：支持 #参数名.属性
    MethodBasedEvaluationContext context = new MethodBasedEvaluationContext(
            null, signature.getMethod(), args, PARAM_NAME_DISCOVERER);

    // 解析 "'seckill:' + #request.activityId + ':' + #authentication.principal"
    return SPEL_PARSER.parseExpression(keyExpr).getValue(context, String.class);
    // 返回 "seckill:1:42"
}
```

**为什么能通过 `#request.activityId` 引用方法参数？**
1. `pom.xml` 中开启了 `<parameters>true</parameters>` 编译选项
2. 编译后的 class 文件保留了方法参数名（`request`, `authentication`）
3. `StandardReflectionParameterNameDiscoverer` 能读取这些名字
4. `MethodBasedEvaluationContext` 将参数名与值绑定，供 SpEL 使用

### 4.4 Redis 令牌桶 Lua 脚本

这是整个限流的核心，用 Lua 脚本保证 **原子性**（Redis 单线程执行 Lua，无需加锁）：

```lua
local key = KEYS[1]        -- rate_limit:seckill:1:42
local rate = tonumber(ARGV[1])   -- qps = 5
local cap = tonumber(ARGV[2])    -- capacity = 10
local now = tonumber(ARGV[3])    -- 当前时间戳(毫秒)

local tokens_key = key .. ':tokens'  -- 存储剩余令牌
local last_key   = key .. ':last'    -- 上次更新时间

local last = redis.call('GET', last_key)

if last == false then
    -- 首次访问：桶满，立即消耗 1 个令牌
    redis.call('SET', tokens_key, cap - 1)
    redis.call('SET', last_key, now)
    redis.call('EXPIRE', tokens_key, math.ceil(cap / rate) + 1)
    redis.call('EXPIRE', last_key, math.ceil(cap / rate) + 1)
    return 1  -- 放行
end

-- 读取当前令牌数
local tokens = tonumber(redis.call('GET', tokens_key))
if tokens == nil then tokens = cap end

-- 计算时间差，补充令牌
local elapsed = math.max(0, now - last)
local new_tokens = math.floor(elapsed * rate / 1000)
tokens = tokens + new_tokens
if tokens > cap then tokens = cap end

if tokens < 1 then
    return 0  -- 令牌用完，限流
end

-- 消耗 1 个令牌
tokens = tokens - 1
redis.call('SET', tokens_key, tokens)
redis.call('SET', last_key, now)
redis.call('EXPIRE', tokens_key, math.ceil(cap / rate) + 1)
redis.call('EXPIRE', last_key, math.ceil(cap / rate) + 1)

return 1  -- 放行
```

**为什么要用 Lua？**

| 方式 | 问题 |
|------|------|
| 先 GET 再 SET（两步操作） | 非原子，并发时出现超卖（double spend） |
| **Lua 脚本** | ✅ **Redis 单线程执行整个脚本，原子性有保证** |

### 4.5 异常处理（`RateLimitException` + `GlobalExceptionHandler`）

```java
// 自定义异常
public class RateLimitException extends BizException {
    public RateLimitException(String message) {
        super(429, message);  // 429 = Too Many Requests
    }
}

// 全局捕获
@ExceptionHandler(RateLimitException.class)
@ResponseStatus(TOO_MANY_REQUESTS)  // HTTP 429
public R<Void> handleRateLimit(RateLimitException e) {
    log.warn("限流触发: {}", e.getMessage());
    return R.fail(TOO_MANY_REQUESTS.value(), e.getMessage());
}
```

**请求链路：**

```
RateLimitAspect 抛出 RateLimitException
    ↓
GlobalExceptionHandler @ResponseStatus(429)
    ↓
Spring Boot 自动设置 HTTP Status 429
    ↓
前端收到 429 → 显示 "秒杀太火爆啦，请稍后再试"
```

## 五、Redis 中的数据变化

以 `qps=5, capacity=10` 为例，某个用户/活动维度的 Redis key 变化：

```
时间轴   请求    令牌桶               Redis 中的 key
───┼───  ─────  ─────────────────  ────────────────────────────
   │   T=0   初始                      (不存在)
   │   T=0   请求 #1  桶: 10→9       rate_limit:1:42:tokens=9
   │                                 rate_limit:1:42:last=0
   │
   │   T=0   请求 #2  桶: 9→8        tokens=8
   │   T=0   请求 #10 桶: 1→0        tokens=0
   │   T=0   请求 #11 桶: 空 → 限流   return 429
   │
   │   T=200ms   (自动补充 0 个，每 200ms 补 1 个)
   │   T=200ms 请求 #11 桶: 1→0       通过
   │
   │   T=500ms  (自动补充 2 个)
   │   T=500ms 请求 #12,#13 通过
   │
   │   T=1s     (自动补充 5 个)
   │   T=1s    请求 #14-18 通过
```

**关键点：**
- 前 10 个请求瞬间通过（用光 capacity）
- 第 11 个请求被限流（429）
- 200ms 后补充 1 个令牌，第 11 个请求放行
- 1 秒后补充 5 个令牌，继续放行

## 六、技术要点汇总

### 6.1 为什么选择 Redis 而不是本地内存？

| 方案 | 优点 | 缺点 |
|------|------|------|
| **本地内存** (Guava RateLimiter) | 快，零网络开销 | ❌ 多实例不共享，每台服务器各自限流 |
| **Redis 令牌桶** | ✅ 全局一致、原子操作、持久化 | 增加一次网络调用（但毫秒级） |

**生产环境多实例部署时，本地限流会失效**（每台服务器的限制是独立的，总 QPS 变成 N 倍）。
Redis 方案确保所有实例共享同一个令牌桶。

### 6.2 为什么用 Redis Lua 而不是 Java 代码计算？

```
Java 实现：
  从 Redis GET tokens → Java 计算 → SET tokens 回 Redis
  → 并发时两个线程同时 GET 到 tokens=10 → 都减 1 → 实际消耗 2 个名额

Lua 实现：
  Redis 单线程执行整个脚本
  → 原子性：读取→计算→写入 一气呵成
  → 不会出现并发超卖
```

### 6.3 为什么用 SpEL 而不用硬编码 key？

```
硬编码：key = "seckill:" + activityId + ":" + userId
  → 每个需要限流的方法都要手动拼接字符串
  → 无法复用，容易出错

SpEL：key = "'seckill:' + #request.activityId + ':' + #authentication.principal"
  → 直接从方法参数提取
  → 支持级联属性、方法调用、字符串运算
  → 注解声明式，清晰直观
```

### 6.4 编译选项 `-parameters` 的作用

```xml
<plugin>
    <groupId>org.apache.maven.plugins</groupId>
    <artifactId>maven-compiler-plugin</artifactId>
    <configuration>
        <parameters>true</parameters>  <!-- 保留方法参数名 -->
    </configuration>
</plugin>
```

没有 `-parameters`，反射获取的参数名是 `args0`, `args1`...，SpEL 无法通过 `#request` 引用。
加上后，参数名变成真正的 `request`, `authentication`。

## 八、总结

```
用户请求 POST /seckill/execute
    ↓
@RateLimit 注解被 AOP 拦截
    ↓
SpEL 解析 key → "seckill:1:42"
    ↓
Redis Lua 令牌桶脚本
    ├── 有令牌 → 放行 → 执行秒杀业务 → 200 OK
    └── 无令牌 → 拒绝 → 抛出异常 → 429 Too Many Requests
```

**核心要点：**
1. **@RateLimit 注解** — 声明式限流，一行注解搞定
2. **AOP 切面** — 与业务代码解耦，限流逻辑集中管理
3. **Redis 令牌桶** — 全局一致、原子操作、支持多实例
4. **Lua 脚本** — 保证并发安全，不超卖
5. **SpEL key** — 灵活定义限流维度（按用户、按活动、按 IP 等）
6. **按用户+活动维度** — 每个用户每个活动独立限流，公平竞争

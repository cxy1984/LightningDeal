# 📸 LightningDeal 图片存储与展示流程

## 一、什么是 MinIO？

**MinIO** 是一个高性能、兼容 Amazon S3 协议的对象存储系统。简单理解，它是一个"自己搭建的云存储"，类似阿里云 OSS、腾讯云 COS，但你可以部署在自己的服务器上。

```
MinIO = 私有化的阿里云 OSS
```

### 在本项目中的角色

| 作用 | 说明 |
|------|------|
| **商品图片存储** | 所有秒杀活动的商品图片都存于 MinIO |
| **文件上传/下载** | 提供 S3 兼容的 API，用于上传和访问文件 |
| **独立于应用** | MinIO 是一个独立的 Docker 容器，与 Spring Boot 后端分离 |

## 二、项目中的 MinIO 配置

### 2.1 Docker Compose 配置 (`infra/docker-compose.yml`)

```yaml
minio:
  image: minio/minio:latest          # Docker Hub 上的官方镜像
  container_name: ld-minio
  environment:
    MINIO_ROOT_USER: minioadmin      # 管理员用户名
    MINIO_ROOT_PASSWORD: minioadmin   # 管理员密码
  command: server /data --console-address ":9001"  # API:9000, 控制台:9001
  ports:
    - "9000:9000"    # S3 API 端口（后端和前端通过此端口访问）
    - "9001:9001"    # Web 管理控制台端口（浏览器访问）
  volumes:
    - minio-data:/data   # 持久化存储，重启容器数据不丢失
```

### 2.2 Spring Boot 配置 (`application.yml`)

```yaml
minio:
  endpoint: http://localhost:9000     # MinIO 服务地址
  access-key: minioadmin              # 访问密钥
  secret-key: minioadmin              # 秘密密钥
  bucket: lightning-deal             # 存储桶名称（类似文件夹）
```

### 2.3 MinIO Java 客户端配置 (`MinIOConfig.java`)

```java
@Bean
public MinioClient minioClient() {
    return MinioClient.builder()
            .endpoint(endpoint)         // http://localhost:9000
            .credentials(accessKey, secretKey)  // minioadmin / minioadmin
            .build();
}
```

启动时自动创建存储桶 `lightning-deal`（如果不存在）。

## 四、详细步骤说明

### 🔼 图片上传流程

#### 第 1 步：上传请求

前端或管理员通过 `POST /api/file/upload` 接口上传图片文件（multipart/form-data）。

```javascript
// 前端 API 调用
const formData = new FormData()
formData.append('file', fileBlob)
api.uploadFile(formData)  // POST /api/file/upload
```

#### 第 2 步：后端接收并存入 MinIO

`FileService.upload()` 方法：

```java
public String upload(MultipartFile file) {
    // 生成唯一文件名，防止覆盖
    String filename = UUID.randomUUID().toString() + "_" + file.getOriginalFilename();

    // 上传到 MinIO 的 lightning-deal 存储桶
    minioClient.putObject(PutObjectArgs.builder()
            .bucket("lightning-deal")     // 存储桶名
            .object(filename)             // 文件名
            .stream(inputStream, size, -1) // 文件流
            .contentType(file.getContentType())  // MIME 类型
            .build());

    // 生成并返回可访问的图片 URL
    return getFileUrl(filename);
}
```

#### 第 3 步：获取图片访问 URL（核心）

```java
public String getFileUrl(String filename) {
    // 生成一个 7 天有效的预签名 URL（Presigned URL）
    return minioClient.getPresignedObjectUrl(GetPresignedObjectUrlArgs.builder()
            .bucket("lightning-deal")
            .object(filename)
            .method(Method.GET)
            .expiry(7, TimeUnit.DAYS)  // 7 天有效期
            .build());
}
```

**预签名 URL（Presigned URL）原理：**

> 预签名 URL 是 MinIO/S3 提供的一种安全机制。它在 URL 中加入签名参数（`X-Amz-Algorithm`、`X-Amz-Credential`、`X-Amz-Signature` 等），使得**没有访问权限的用户也能在有效期内通过该 URL 访问文件**。
>
> 相当于 MinIO 给你发了一张"7 天内有效的临时门票"。

**示例 URL：**
```
http://localhost:9000/lightning-deal/xxx_iphone.jpg?X-Amz-Algorithm=AWS4-HMAC-SHA256&X-Amz-Credential=minioadmin%2F20260629...&X-Amz-Signature=xxx...
```

#### 第 4 步：将 URL 存入数据库

上传返回的图片 URL 需要在创建/编辑活动时，存入 `seckill_activity.goods_image` 字段，才能在后端返回数据时一并返回。

```sql
-- 字段定义
goods_image VARCHAR(1024) COMMENT '商品图片URL'

-- 更新活动图片
UPDATE seckill_activity SET goods_image = 'http://localhost:9000/lightning-deal/xxx.jpg' WHERE id = 1;
```

### 🔽 图片展示流程

#### 第 5 步：前端请求活动列表

```javascript
// 前端调用
api.getActivityList({ page: 1, size: 12 })
// → GET /api/activity/list?page=1&size=12
```

#### 第 6 步：后端从 MySQL 查询并返回

`SeckillActivityServiceImpl.toVO()` 方法：

```java
private ActivityVO toVO(SeckillActivity activity) {
    ActivityVO vo = new ActivityVO();
    BeanUtils.copyProperties(activity, vo);  // 复制所有字段，包括 goodsImage
    // ... 其他业务逻辑
    return vo;
}
```

返回的 JSON 中包含 `goodsImage` 字段：
```json
{
  "id": 1,
  "name": "iPhone 15 Pro 限时秒杀",
  "goodsImage": "http://localhost:9000/lightning-deal/xxx_iphone.jpg",
  "flashPrice": 6999.00,
  ...
}
```

#### 第 7 步：前端 `<el-image>` 加载图片

```html
<el-image :src="item.goodsImage || '/placeholder.png'" fit="cover">
  <template #error>
    <div class="image-placeholder">📷 {{ item.goodsName }}</div>
  </template>
</el-image>
```

- 如果 `goodsImage` 有值 → 浏览器从 MinIO 加载图片
- 如果 `goodsImage` 为空 → 显示占位符 `/placeholder.png`
- 如果图片加载失败 → 显示 📷 + 商品名称

#### 第 8 步：浏览器直接请求 MinIO

浏览器根据 `<img>` 标签的 `src` 属性，直接向 `http://localhost:9000`（MinIO）发起 GET 请求。

**为什么能访问？**
- ✅ MinIO 的 S3 API 端口（9000）被 `docker-compose` 映射到宿主机
- ✅ 我们已将 `lightning-deal` 存储桶设为 **匿名可读**（`mc anonymous set download`）
- ✅ 或者使用预签名 URL（带有时效的临时访问令牌）

## 五、关键技术原理

### 5.1 为什么用 MinIO 而不是直接存 MySQL？

| 存储方式 | 优点 | 缺点 |
|---------|------|------|
| **MySQL BLOB** | 简单，直接存在数据库里 | ❌ 数据库膨胀、备份慢、读写性能差 |
| **文件系统** | 简单 | ❌ 分布式部署难以共享、备份复杂 |
| **MinIO (对象存储)** | ✅ 高性能、分布式、S3 标准 | 需要额外部署和维护 |

**本项目选择 MinIO 的原因：**
1. **高性能**：图片文件通过 HTTP 直接访问，不经过后端，减少后端负载
2. **S3 标准**：可以无缝迁移到阿里云 OSS、腾讯云 COS 等云服务
3. **独立部署**：作为独立服务运行，与 Java 后端解耦
4. **安全性**：支持预签名 URL、存储桶策略、访问密钥等安全机制

### 5.2 预签名 URL 原理

```
预签名 URL = 文件路径 + 签名参数（有效期、访问密钥、算法等）
              ↓
浏览器拿着这个 URL 请求 MinIO
              ↓
MinIO 验证签名是否有效且在有效期内
              ↓
有效 → 返回图片   /   无效 → 拒绝访问
```

**为什么不用直接的 URL？（我们后来改用了直接 URL）**

因为在开发环境中，我们已经将存储桶设为匿名可读，所以可以直接用 `http://localhost:9000/bucket/filename` 访问，无需签名。

生产环境中，应该使用预签名 URL 或更严格的存储桶策略，防止文件被随意访问。

### 5.3 CORS 跨域问题

前端 `localhost:5173` 请求 MinIO `localhost:9000` 属于**跨域请求**（端口不同）。

解决方案：
```
MinIO 配置允许来自前端域名的跨域请求
AllowedOrigins: http://localhost:5173, http://localhost:8080
```

### 5.4 Docker Compose 中的网络通信

```
┌───────────────┐      docker network      ┌──────────────┐
│  Spring Boot  │──────────────────────────>│    MinIO     │
│  (ld-backend) │  http://minio:9000        │  (ld-minio)  │
│               │  （容器内通信）             │              │
│  宿主机端口    │                           │  宿主机端口   │
│  8080         │                           │  9000        │
└───────────────┘                           └──────────────┘
       ↑                                            ↑
       │  前端通过 localhost:8080                    │  浏览器通过 localhost:9000
       │  （Vite 代理）                              │  （直接请求）
       │                                            │
┌──────┴──────┐                                    │
```

## 六、项目中的实际操作

### 6.1 启动 MinIO

```bash
cd infra
docker compose up -d minio
```

### 6.2 设置存储桶为公开读（开发环境）

```bash
docker exec ld-minio mc alias set local http://localhost:9000 minioadmin minioadmin
docker exec ld-minio mc anonymous set download local/lightning-deal
```

### 6.3 上传图片

```bash
curl -X POST http://localhost:8080/api/file/upload \
  -H "Authorization: Bearer <token>" \
  -F "file=@图片路径"
```

返回：
```json
{"code":200,"msg":"上传成功","data":"http://localhost:9000/lightning-deal/xxx.jpg"}
```

### 6.4 关联活动

```sql
UPDATE seckill_activity 
SET goods_image = 'http://localhost:9000/lightning-deal/xxx.jpg' 
WHERE id = 1;
```

### 6.5 前端展示

刷新页面，`<el-image>` 组件自动加载图片。

## 七、总结

```
用户上传图片
    ↓
后端接收 → 存入 MinIO → 返回 URL
    ↓
URL 存入 MySQL（关联到活动）
    ↓
前端请求活动列表
    ↓
后端返回数据（含 goodsImage URL）
    ↓
前端 <el-image> 从 MinIO 加载图片
    ↓
图片显示在浏览器中 ✅
```

**核心要点：**
1. **MinIO** 是图片的最终存放地，类似"私有云盘"
2. **后端** 充当"搬运工"，负责接收上传、存入 MinIO、返回 URL
3. **MySQL** 只存 URL 字符串，不存图片文件本身
4. **前端** 直接请求 MinIO 获取图片，不经过后端，减轻后端压力
5. **预签名 URL** 提供安全的临时访问控制

# LightningDeal 压测记录（2026-07-09）

## 1. 压测目标

本轮压测主要验证秒杀系统在并发请求下的核心稳定性：

| 验证项 | 目标 |
|------|------|
| 限流是否生效 | 高并发下大部分请求应被 429 拦截 |
| 库存是否超卖 | Redis 预扣 + MySQL 兜底后，最终库存不能为负 |
| MQ 消费能力 | 成功进入队列的请求应在秒级完成异步下单 |
| 响应延迟 | P95/P99 延迟应保持可接受范围 |
| 错误率 | 不应出现大量 5xx/超时错误 |

## 3. 压测前发现的问题

### 3.1 原限流 key 是按用户隔离的

原代码：

```java
@RateLimit(
    qps = 5,
    capacity = 10,
    key = "'seckill:' + #request.activityId + ':' + #authentication.principal"
)
```

问题：

- `activityId + userId` 作为 key，意味着每个用户都有独立令牌桶。
- 多用户压测时，198 个用户会有 198 个桶。
- 结果：所有请求都能通过，看起来限流没有生效。

压测结果表现：

| 指标 | 数值 |
|------|------|
| 实际请求 | 198 |
| 成功 200 | 198 |
| 限流 429 | 0 |
| QPS | 482 |
| 库存 | 198 + 302 = 500，未超卖 |

结论：**按用户限流适合防止单用户刷接口，但不能保护系统整体入口。秒杀场景更需要按活动全局限流。**

---

## 4. 限流策略调整

调整为按活动全局限流：

```java
@RateLimit(
    qps = 5,
    capacity = 10,
    key = "'seckill:' + #request.activityId",
    message = "秒杀太火爆啦，请稍后再试"
)
```

调整后含义：

| 配置 | 说明 |
|------|------|
| `qps = 5` | 每秒补充 5 个令牌 |
| `capacity = 10` | 突发最多允许 10 个请求 |
| key = activityId | 同一个活动共享一个令牌桶 |

### 9.4 限流策略建议

当前采用全局活动限流：

```java
key = "'seckill:' + #request.activityId"
```

推荐保留。

如果未来要同时防止单用户刷接口，可以叠加双层限流：

| 层级 | key | 作用 |
|------|-----|------|
| 全局活动限流 | `seckill:{activityId}` | 保护系统入口 |
| 用户级限流 | `seckill:{activityId}:{userId}` | 防止单用户刷接口 |

当前阶段全局限流已经满足秒杀入口保护要求。
